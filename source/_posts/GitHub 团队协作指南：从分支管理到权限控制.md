如果你正在经历以下场景：

- 新同事入职，需要手动给他添加十几个仓库的权限

## 一、为什么要新建 dev 分支并设为默认？

很多团队一开始都是直接在 `main` 分支上开发。这种做法在小规模或个人项目中没问题，但当团队扩大后，问题就来了。

### 1.1 主分支永远要保持可发布

想象一下这个场景：你在开发一个新功能，代码写到一半还不能运行。这时候线上出现了一个紧急 Bug，需要立即修复。如果大家都在 `main` 分支上开发：

- 你的半成品代码还在本地，同事无法帮你继续
- 修复 Bug 的人可能会把你的代码一起带到生产环境

**解决方案**：引入 Git Flow 工作流

```
main 分支（稳定版） 
   ↑
dev 分支（开发版）——功能分支1
                ——功能分支2
                ——Bug修复分支
```



- **main 分支**：永远存放可部署的稳定代码
- **dev 分支**：日常开发集散地，允许有小问题
- **功能分支**：从 dev 拉取，完成后合并回 dev

### 1.2 把 dev 设为默认分支的意义

当把 `dev` 设为默认分支后：

- 新成员克隆仓库时，自动拉取的是最新的开发代码
- Pull Request 默认目标分支就是 `dev`，减少手动修改的错误
- 持续集成工具可以针对 `dev` 分支运行测试，而不影响 `main` 的稳定

### 1.3 实际操作步骤

```
# 1. 创建 dev 分支
git checkout -b dev

# 2. 推送到远程
git push -u origin dev

# 3. 在 GitHub 网页端设置为默认分支
# Settings → Branches → Default branch → 选择 dev → Update
```



## 二、如何管理默认分支的权限？

有了 `dev` 分支后，下一步就是控制谁能往上面推送代码。

### 2.1 权限管理的核心：分支保护规则

GitHub 的权限体系是分层的：

1. **仓库访问权限**：决定谁能看、谁能写
2. **分支保护规则**：精细化控制特定分支

### 2.2 设置分支保护规则（详情见[GitHub分支保护](https://zgj6017.github.io/2026/03/01/GitHub%E5%88%86%E6%94%AF%E4%BF%9D%E6%8A%A4%E4%B8%8ECICD%E5%AE%8C%E5%85%A8%E6%8C%87%E5%8D%97%EF%BC%9A%E4%BB%8E%E4%BB%A3%E7%A0%81%E8%A7%84%E8%8C%83%E5%88%B0%E8%87%AA%E5%8A%A8%E5%8C%96%E9%83%A8%E7%BD%B2/#more))

进入仓库的 `Settings → Branches → Add rule`：

**规则示例 - 开发分支 (dev)**：

- ✅ Require a pull request before merging（必须通过 PR 合并）
- ✅ Require approvals（需要至少 1 人审查）
- ✅ Dismiss stale pull request approvals（新提交后撤销旧批准）

**规则示例 - 主分支 (main)**：

- 以上所有规则
- ✅ Include administrators（管理员也要遵守）
- ✅ Require status checks（必须通过 CI 测试）

### 2.3 限制谁可以推送

在规则中找到 **"Restrict who can push to matching branches"**：

- 可以添加特定用户
- 可以添加整个团队
- 不在列表里的人，即使有写入权限也无法直接推送

## 三、为什么需要团队？如何组建？

随着项目增多，手动管理每个仓库的协作者变得不可持续。这就是 GitHub 组织（Organization）和团队（Team）发挥作用的地方。

### 3.1 团队的价值

- **批量授权**：把一个团队加到仓库，所有成员自动获得权限
- **沟通便利**：用 @团队名 可以一次性通知所有人
- **职责清晰**：前端团队、后端团队、运维团队各司其职

### 3.2 创建组织的步骤

1. 点击 GitHub 右上角 `+` → `New organization`
2. 选择免费计划
3. 填写组织名称（如 `ABC科技`）
4. 邀请初始成员（可选）

### 3.3 创建团队的步骤

进入组织页面 → `Teams` → `New team`：

- **名称**：前端组、后端组、DevOps
- **描述**：让成员知道这个团队的职责
- **可见性**：公开或私密

## 四、如何把团队添加到仓库？

有两种方法，任选其一即可。

### 方法一：从仓库出发（推荐）

1. 进入仓库 → `Settings` → `Collaborators and teams`
2. 点击 `Invite teams or people`
3. 输入团队名称（如 "前端组"）
4. 选择权限级别：

| 权限     | 适用场景           | 能力                 |
| :------- | :----------------- | :------------------- |
| Read     | 产品经理、项目经理 | 查看代码、Issue      |
| Triage   | 测试人员           | 管理 Issue，不能推送 |
| Write    | **开发人员**       | 推送代码、创建 PR    |
| Maintain | 技术负责人         | 管理仓库设置         |
| Admin    | 仓库负责人         | 所有操作权限         |

### 方法二：从团队出发

1. 进入组织 → `Teams` → 选择团队
2. 点击 `Repositories` → `Add repository`
3. 选择仓库并设置权限

## 五、常见误解：团队队长等于仓库所有者？

这是一个需要澄清的重要概念。

### 5.1 两种角色的区别

| 角色           | 权限范围                 | 能否删除仓库 |
| :------------- | :----------------------- | :----------- |
| **团队维护者** | 管理团队成员（增删成员） | ❌ 不能       |
| **仓库管理员** | 管理仓库代码和设置       | ✅ 能         |

### 5.2 举个例子

假设：

- 小张是"前端组"的团队维护者
- "前端组"被添加到 `web-app` 仓库，权限为 `Write`

那么小张可以：

- ✅ 在团队里添加或移除成员
- ✅ 往 `web-app` 推送代码
- ✅ 创建和合并 PR

但小张不能：

- ❌ 修改 `web-app` 的 GitHub 设置
- ❌ 删除 `web-app` 仓库
- ❌ 给其他人分配仓库权限

### 5.3 如何让队长拥有管理权？

如果你确实需要队长管理仓库，需要明确授权：

**方案一**：给团队更高权限

- 在仓库的协作者设置中，把团队的权限改为 `Admin`

**方案二**：单独授权个人

- 把队长作为个人协作者添加，授予 `Admin` 权限

## 六、完整实践：从零搭建一个规范的项目

让我们把以上知识串起来，模拟一个真实场景。

### 场景设定

- 项目名称：`e-commerce`（电商网站）
- 团队组成：前端组（3人）、后端组（4人）、运维组（2人）
- 要求：
  - 只有技术负责人能合并到 `main`
  - 开发人员通过 PR 提交代码到 `dev`
  - 新成员加入自动获得对应权限

### 实施步骤

**第1步：创建组织和团队**

```
ABC科技（组织）
├── 前端组
├── 后端组
├── 运维组
└── 技术委员会（技术负责人）
```



**第2步：创建仓库并设置分支**

```
git init e-commerce
cd e-commerce
git checkout -b main
git checkout -b dev
git push -u origin main dev
```



**第3步：设置默认分支**

- 网页端：`Settings → Branches` → 将 `dev` 设为默认

**第4步：添加分支保护**

dev 分支：

- 必须通过 PR
- 需要 1 个审查
- 允许所有开发人员推送

main 分支：

- 必须通过 PR
- 需要 2 个审查
- 只允许技术委员会推送

**第5步：添加团队到仓库**

- 前端组 → `Write` 权限
- 后端组 → `Write` 权限
- 运维组 → `Read` 权限（或 Triage）
- 技术委员会 → `Admin` 权限

### 成果

- 新来的前端同事：加入"前端组"后，自动获得所有前端项目的 `Write` 权限
- 运维同事：只能查看代码，不能修改
- 技术负责人：拥有所有仓库的最终管理权
- 主分支：永远保持稳定，只有经过审查的代码才能进入

## 七、总结与最佳实践

### 关键原则

1. **分支即环境**：`main` 是生产，`dev` 是开发
2. **保护即规则**：用分支保护规则代替口头约定
3. **团队即权限**：用团队管理代替个人管理
4. **授权即分离**：团队管理和仓库管理是两回事

### 避坑指南

- ❌ 不要给所有人 `Admin` 权限
- ❌ 不要直接往 `main` 推送
- ❌ 不要手动一个个添加协作者
- ✅ 善用团队和分支保护规则

### 后续建议

1. 定期审查仓库权限，移除不必要的访问
2. 培养团队成员通过 PR 协作的习惯
3. 把权限管理文档化，让新成员快速上手
